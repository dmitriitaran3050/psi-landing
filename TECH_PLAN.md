# Технический план MVP: Лендинг-игра `gamification-event`

## Summary

Подготовить локальный `MVP` статического лендинга для проверки гипотезы:

- проходит ли пользователь путь без перегруза
- воспринимается ли формат как психологически безопасный
- ощущаются ли инсайты как персонализированные
- понятен ли следующий шаг в `Telegram`

Техническая рамка:

- `HTML + CSS + JS`
- без фреймворков, сборщика и backend
- локальный запуск и последующий перенос на `GitHub Pages`

Критерии успеха локального `MVP`:

- пользователь проходит сценарий за `2–4 минуты`
- ни один экран не требует пояснений ведущего
- хотя бы один инсайт или итоговый паттерн вызывает реакцию уровня «это похоже на меня»
- переход к `CTA` воспринимается как логичное продолжение сценария

## Scope

В первую версию входят:

- одна ветка: `Отношения`
- стартовый экран
- 3 вопроса
- 3 персонализированных экрана инсайтов
- итоговый экран паттерна
- экран броска кубика
- экран результата приза
- `CTA` на видео и `Telegram`

В первую версию не входят:

- ветки `Я и моя жизнь`, `Деньги и карьера`, `Мягкий вход`
- серверная логика
- `Telegram API`
- учёт лимитов призов
- `A/B` тесты
- прод-аналитика
- админка

## Структура файлов

```text
landing/gamification-event/

├── index.html
```

Ограничения:

- все пути относительные
- код и контент хранятся локально
- зависимости через `npm` не используются

## UI Flow

### 1. `Start`

Экран показывает:

- заголовок
- короткую безопасную рамку без обещаний диагностики
- 4 карточки сфер

Правило:

- активна только карточка `Отношения`
- остальные 3 карточки видимы, помечены `Скоро` и не кликаются

### 2. `Question`

Один экран на один вопрос:

- заголовок вопроса
- 4 варианта ответа
- прогресс `1 из 3`, `2 из 3`, `3 из 3`

Правило:

- выбор ответа сразу фиксируется в `state`
- после выбора открывается отдельный экран инсайта

### 3. `Insight`

Отдельный экран после каждого ответа:

- показывает только тот инсайт, который соответствует выбранному ответу
- содержит кнопку `Продолжить`
- автопереход не используется

### 4. `Pattern Result`

Экран показывает:

- название паттерна
- краткое описание
- `CTA` `Посмотреть видео`
- `CTA` `Перейти в Telegram`
- отдельную кнопку `Бросить кубик`

Рамка:

- это тёплая интерпретация
- это не диагноз
- это не терапия и не её замена

### 5. `Dice Intro`

Экран завершения сценария:

- подведение к ритуалу завершения
- кнопка запуска броска

### 6. `Prize Result`

Экран показывает:

- выпавшую грань
- название приза
- описание приза
- `CTA` `Получить приз в Telegram`
- кнопку `Пройти заново`

## Контентные ограничения

Запрещено:

- язык диагноза
- обещания точной психологической диагностики
- эзотерическая лексика
- манипулятивные обещания
- подача результата как клинического вывода

Разрешённая рамка:

- `первый инсайт`
- `паттерн внимания`
- `гипотеза о повторяющемся сценарии`
- `точка для самонаблюдения`

## Данные и логика

Контент хранится в одном объекте `APP_DATA`.

Минимальная структура:

- `categories`
- `flows.relationships.questions`
- `flows.relationships.patterns`
- `flows.relationships.resultRule`
- `prizes`
- `links`

Минимальные поля:

- `category.id`, `title`, `subtitle`, `icon`, `enabled`, `badge`
- `question.id`, `title`, `progressLabel`, `options`
- `option.id`, `label`, `insightTitle`, `insightText`
- `pattern.id`, `title`, `description`
- `prize.id`, `diceFace`, `title`, `description`
- `links.videoUrl`, `links.telegramUrl`

Правило расширяемости:

- новые ветки добавляются через данные
- `renderQuestion`, `renderInsight`, `renderPattern`, `renderPrize` не должны зависеть от конкретной ветки

## Правило вычисления паттерна

В `MVP` у ветки `Отношения` есть один итоговый паттерн.

Правило:

- итог не зависит от комбинации ответов
- персонализация достигается через отдельные инсайты после каждого ответа

Причина:

- упрощение первой версии
- отсутствие ложной имитации психологической диагностики
- быстрое тестирование сценария и контента

## State Management

Единый объект состояния:

- `screen`
- `selectedCategoryId`
- `currentQuestionIndex`
- `selectedAnswers`
- `currentInsight`
- `patternId`
- `diceRolled`
- `prizeId`
- `isTransitioning`

Правила:

- одновременно активен только один экран
- повторные клики во время переходов и анимации блокируются
- состояние после перезагрузки не сохраняется
- кнопка `Пройти заново` делает полный `reset`

## Механика кубика

Правила:

- честный `random` от `1` до `6`
- все 6 граней равновероятны
- повторный бросок в одной сессии невозможен
- повторное прохождение возможно только через `reset`

Поведение:

- нажатие `Бросить`
- анимация `1–1.5s`
- блокировка интерфейса на время анимации
- после анимации рендер результата приза

Ограничения:

- лимиты призов автоматически не контролируются
- описание призов статическое
- выдача приза вне сайта не реализуется

## CTA и ссылки

На экране паттерна:

- `Посмотреть видео`
- `Перейти в Telegram`

На экране приза:

- `Получить приз в Telegram`

Правило для локальной разработки:

- используются рабочие тестовые `URL`
- кнопки не ведут в пустоту
- если ссылка ещё не готова, рядом показывается пояснение

## UX и визуальная рамка

Требования:

- `mobile-first`
- управление одной рукой
- крупные `tap-target`
- короткие, спокойные анимации
- современный и тёплый визуальный стиль
- кубик воспринимается как игровой ритуал завершения, а не как мистический символ

Нельзя использовать:

- неон
- космическую или таро-эстетику
- «магические» свечения
- длинные декоративные задержки
- симуляцию сложной аналитики

## Аналитика как точка расширения

Внешняя аналитика в `MVP` не внедряется.

События, которые нужно предусмотреть:

- `start_selected_relationships`
- `question_answered`
- `insight_continued`
- `pattern_viewed`
- `cta_video_clicked`
- `cta_telegram_clicked`
- `dice_started`
- `prize_viewed`
- `restart_clicked`

Реализация:

- функция `track(eventName, payload)`
- в `MVP` она пишет события в `console.debug`

## Test Plan

### Функциональные проверки

- стартовый экран показывает 4 карточки, кликабельна только `Отношения`
- на каждом вопросе доступно 4 ответа
- после выбора ответа показывается только связанный с ним инсайт
- кнопка `Продолжить` переводит к следующему вопросу или к экрану результата
- после третьего инсайта открывается экран паттерна
- на экране паттерна работают оба `CTA`
- бросок кубика доступен один раз за прохождение
- экран приза показывает одну грань и один приз
- `Пройти заново` полностью сбрасывает состояние

### UX-проверки

- сценарий проходится одной рукой на мобильном
- тексты читаются без ощущения перегруза
- инсайты успевают прочитать без спешки
- пользователь понимает, что это сценарий саморефлексии, а не тест
- кубик воспринимается как завершение, а не как допуск

### Edge Cases

- быстрые повторные клики не ломают интерфейс
- во время анимации кубика действия не дублируются
- повторный запуск после `reset` работает корректно
- страница стабильно работает в мобильных `Safari` и `Chrome`

## Assumptions

- `MVP` предназначен для локальной сборки и тестирования
- итоговый паттерн один
- основная персонализация строится через инсайты на ответы
- расширение до 4 веток переносится на следующий этап
